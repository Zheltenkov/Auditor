## Chapter III <div id="chapter-iii"></div>

### 1. Theory <div id="31"></div>

REST (Representational State Transfer, [Рой Филдинг](https://en.wikipedia.org/wiki/Roy_Fielding) 2000 г.) является архитектурным стилем для обеспечения стандартов взаимодействия между компьютерными системами в сети. Следование стандартам облегчает для систем обмен данными друг с другом. Системы, отвечающие требованиям REST и часто называются RESTful. 

Архитектура RESTful-интеграции должна придерживаться следующих 6 принципов (ограничений Филдинга):

1. Клиент-серверная архитектура: разделение зон ответственности, функции клиента и сервера разделены. 
2. Stateless: сервер в каждом запросе получает всю информацию для обработки (не сохраняет информацию у себя).
3. Кэширование: сохранение ответов сервером для дальнейшего использования.
4. Единообразие интерфейса (HATEOAS): ограничения стиля, по которому сервер возвращает не только ресурс, но и его связи с другими ресурсами и действия, которые можно с ним совершить.
5. Слоистая архитектура (Layered system): ни клиент, ни сервер не знают о том, как происходит цепочка вызовов дальше их прямых контактов.
6. Код по требованию (Code on demand): передача исполняемого кода (необязательная опция).

Более подробно с принципами и свойствами архитектурного стиля REST, преимуществами и недостатками подхода можно познакомиться в статье А. Буракова «REST, что же ты такое? Понятное введение в технологию для ИТ-аналитиков».

### 2. Основные понятия <div id="31"></div>

**Формат представления данных**: способ, которым передаются данные, например:

- XML — язык разметки информации;
- JSON — текстовый формат обмена данными;
- Binary — бинарный формат.

**Протокол передачи данных** — совокупность соглашений, определяющих порядок обмена данными между системами и/или компонентами систем. Соглашения задают:

- правила обмена;
- набор форматов передачи данных;
- единый способ передачи сообщений;
- порядок обработки ошибок.

**Транспорт** — протоколы передачи данных по сети.

**Ресурсы: https://21-school.ru/blog**

**https://** — scheme, указание транспорта, протокола.

**21-school.ru** — host, сервер, с которым идет взаимодействие. 

**blog** — ресурс, объект, с которым выполняется взаимодействие, блог Школы 21.

**HTTP-протокол**: HTTP — широко распространённый протокол передачи данных, изначально предназначенный для передачи гипертекстовых документов (то есть документов, которые могут содержать ссылки, позволяющие организовать переход к другим документам). Используется и для передачи произвольных данных.

**HTTP-глаголы**

В RESTful-интеграции операции описываются HTTP-глаголами.

Существует 4 основных метода HTTP, которые используются в запросах для взаимодействия с ресурсами в системе REST:

GET — получение конкретного ресурса (по ID) или коллекцию ресурсов;

POST — создание нового ресурса;

PUT — обновление конкретного ресурса (по ID);

DELETE — удаление конкретного ресурса по ID.

**API** — это:

- Application Programming Interface (программный интерфейс приложения), описание способа взаимодействия программных систем или компонентов систем;
- Механизм взаимодействия программных компонент;
- Продукт, позволяющий предоставлять доступ внешним пользователям к системе.

API — контракт (соглашение, договор, описание) взаимодействия между программами. Содержит:

- операцию, которую должна (и может) выполнить система;
- входные данные (данные, которые поступают на вход);
- выходные данные (данные, которые можно получить по результатам операции);
- сообщения об ошибках (описание ошибок).

### 4. Маппинг <div id="34"></div>

Одна из наиболее важных работ аналитика на интеграции — маппинг данных с разных сторон интеграции. Данные в системах, которые взаимодействуют между собой, и в данных, которые передаются, могут храниться в разных структурах. Например, в одной системе ФИО — единое обязательное поле, в другой — это три поля, и только два из них обязательны, а отчество может не заполняться, а в третьей системе обязательны все три поля. Или при отражении денежных сумм: при передаче (чтении/записи) денежных данных в числовом формате возможны искажения при округлениях.

Маппинг — сопоставление полей данных в разных источниках хранения и передачи данных. Учти, что сопоставление данных требуется практически при любом типе обмена.

Что следует сопоставлять и учитывать при сопоставлении:

1. Идентификатор — системное наименование поля, под которым данные хранятся в источнике или передаются в сообщении. В случае хранения поля в таблице — запись через точку (идентификатор таблицы.идентификатор поля). Либо название таблицы выносится выше, как оглавление группы полей.
2. Бизнесовое наименование/описание поля.
3. Тип данных (текст, число (целое или дробное), дата, список значений, пр.).
4. Размерность (длина поля).
5. Обязательность: может быть разной для системы-отправителя и системы-получателя.
6. Комментарии: условия, ограничения, порядок обработки/преобразований при необходимости.

### 5. Обработка ошибок <div id="35"></div>

Система-получатель принимает сообщения, сформированные другой системой. Не всегда система-отправитель формирует сообщение так, как оговорено в контракте. Поэтому при получении сообщения система-получатель должна проверять получаемое сообщение. И в случае получения некорректных сообщений следует сообщить системе-отправителю о проблеме. Такие сообщения система может отправить в ответе (в поле сообщения об ошибке). Описание о полученных несоответствиях, как правило, готовит аналитик, поскольку хорошо знает, что следует проверять в полученном сообщении. Проверяются требования к формату (тип данных, размерность, обязательность), наличие в справочнике (если это ссылка на запись в справочнике), другие логические условия.

Кроме того, в сообщениях об ошибках передают и другие сообщения, например, о недоступности системы-получателя.   

## Chapter IV <div id="chapter-iv"></div>

### Description of tasks

### Задача 1. Запись на стрижку (Sign up for a haircut) <div id="41"></div>

Руководство сети барбершопов приняло решение о внедрении системы, обеспечивающей онлайн-запись на прием. Основная цель — развитие бизнеса путем расширения клиентской базы за счет возможности онлайн-записи, а также снижение трудозатрат сотрудников и уменьшение ручного труда за счет автоматического информирования клиентов по каналам связи.

Запись может осуществлять как зарегистрированный, так и незарегистрированный посетитель сайта. При записи можно выбрать услугу, мастера и время из свободных интервалов. Система должна обеспечивать автоматическую отправку напоминаний клиентам через выбранный клиентом канал связи (Telegram, WhatsApp, VK, СМС) по настроенному менеджером расписанию. После получения услуги система предлагает клиенту оценить услугу и написать предложения по улучшению работы.

Расписание мастеров и выполняемые каждым мастером услуги должен вводить менеджер, возможно, это будет не один человек. Он же отвечает за актуальность расписания и при необходимости корректирует его, осуществляет связь с клиентами в ручном режиме, проставляет отметку о выполнении услуги, начисляет и принимает оплату, передает данные об оплате в бухгалтерию. Также менеджер может получать отчеты о выполненных услугах и просматривать отзывы клиентов.

Любой мастер имеет возможность посмотреть расписание и запись на свои услуги, отзывы клиентов.

**Сокращения**

Слот: период времени длительностью 1 час, указано начальное время.

**UC02. Сформировать расписание мастера**

**Контекст использования:** менеджер создает расписание работы мастера в виде слотов для выбора клиентами.

**Область действия:** барбершоп, оказание услуг клиентам.

**Уровень:** цель пользователя.

**Основное действующее лицо:** менеджер.

**Предусловия:** Менеджер получил график работы мастеров. Менеджер зашел на сайт.

**Основной сценарий:**

1. Менеджер открывает страницу ввода слотов мастеров на обслуживание. 
2. Менеджер выбирает дату и слот (период времени длительностью 1 час, указано начальное время), выбирает мастера из сущности «Сотрудники», которым указана роль «Мастер» в сущности «Роль сотрудника».
3. Система подтверждает, что выбранный слот не забронирован в системе клиентом.
4. Менеджер подтверждает закрепление времени за мастером.
5. Система закрепляет слот: дата, время, мастер.

**Альтернативный сценарий:**

1. Выбранный слот забронирован в системе клиентом:
   1. Система сообщает, что выбранный слот забронирован в системе клиентом.
   2. Менеджер связывается по телефону с клиентом и договаривается о переносе визита на другое время или к другому мастеру, вносит информацию в систему.
   3. Система направляет СМС по телефону клиента об изменении времени и/или мастера визита.

**UC04. Зарегистрировать клиента (менеджер)**

**Контекст использования:** Менеджер, при общении с клиентом по телефону, регистрирует клиента, чтобы записать на обслуживание.

**Область действия:** барбершоп, оказание услуг клиентам.

**Уровень:** цель пользователя.

**Основное действующее лицо:** менеджер.

**Предусловия:** Клиент позвонил менеджеру. Менеджер зашел на сайт барбершопа.

**Основной сценарий:**

1. Менеджер открывает страницу регистрации клиента. 
2. Менеджер вводит номер телефона и имя, которое клиент сообщил по телефону.
3. Система подтверждает, что телефон не зарегистрирован в системе.
4. Система регистрирует клиента, генерирует временный пароль, направляет СМС по телефону клиента о регистрации с указанием имени, логина, временного пароля.

**Альтернативный сценарий:**

1. Введенный телефон зарегистрирован в системе:
   1. Система сообщает, что такой номер телефона зарегистрирован в системе.
   2. Клиент вводит другой номер телефона (2.) или отказывается от регистрации.

2) Согласованность движений, действий двух или более людей, явлений или процессов.
3) Архитектурный стиль взаимодействия компонентов распределенного приложения в сети.
4) Протокол взаимодействия компонентов распределенного приложения, предложенный Роем Филдингом.
5) Способность системы, сети или процесса увеличивать свою производительность при добавлении ресурсов.
6) Свойство метода, позволяющее при повторном вызове приводить сервис в то же состояние, что и первичный вызов.
7) Способность системы, сети или процесса увеличивать свою производительность без добавления ресурсов.
8) Увеличение производительности веб-приложений за счёт использования сохранённых ранее данных.
9) Синдром информационной усталости.
